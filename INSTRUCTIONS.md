# HRMS Build — Agent Execution Brief

## 0. Role & Operating Mode

You are a senior Full-Stack Engineer and System Architect building a production-grade Human Resource Management System (HRMS) end to end.

You will work **incrementally, phase by phase** (see Section 11). You will not attempt to generate the entire system in one pass. You will not move to the next phase until the current phase's Definition of Done is fully satisfied.

You maintain your own state across the build using four scaffold files, created before Phase 0 and updated after every phase:

- `AGENTS.md` — your operating rules for this project (copy the relevant sections of this brief into it, don't just link to it)
- `CONTEXT.md` — architecture decisions made so far, and why (append-only log)
- `TASK.md` — the current phase's task list, checked off as completed
- `PROGRESS.md` — what's done, what's pending, what was assumed (Section 14) and why

This exists so that a fresh context window (yours in a later turn, or another agent) can resume the build without re-deriving decisions or silently re-deciding them differently.

## 1. Source of Truth & Conflict Resolution

Priority order, highest first:

1. This brief's explicit specifications (Sections 2, 5–9)
2. The original SRS intent (feature list, roles, modules)
3. Wireframes — visual/layout inspiration only, never authoritative for behavior or data model
4. Your own judgment — only when none of the above cover the case, and only after logging the assumption in `PROGRESS.md`

If a decision would change data model shape, security behavior, or role permissions, **do not silently assume** — flag it in `PROGRESS.md` and proceed with the documented default in Section 2 if one exists.

## 2. Non-Negotiable Defaults

These resolve every ambiguity in the original spec. Do not deviate without flagging why.

**Accounts & roles**
- Self-registration always creates an `EMPLOYEE` role account. There is no self-service path to `ADMIN`/`HR`. Admin accounts are created only by an existing Admin via an authenticated admin-only endpoint, or via the seed script for the first bootstrap account.
- The Login ID should be automatically generated by the system in the following format:
[OI (first two letters of the employee’s first and last name) (year of joining) (serial number of joining)]

Example: OIJODO20220001

Explanation:

OI → Odoo India (Company Name)
JODO → First two letters of the employee’s first name and last name
2022 → Year of Joining
0001 → Serial Number of Joining for that Year

**Tokens & sessions**
- Access token: JWT, 15 minute expiry, sent in `Authorization: Bearer`.
- Refresh token: opaque random string (not JWT), 7 day expiry (30 days if "Remember Me"), stored **hashed** in the `RefreshToken` table, delivered as an `httpOnly`, `secure`, `sameSite=strict` cookie — never in localStorage.
- Refresh rotation: every refresh call issues a new refresh token and invalidates the old one (rotation, not reuse). Reuse of an already-rotated token revokes the entire token family for that user (theft detection).
- Email verification token: 24 hour expiry, single-use, resend endpoint allowed with a 60-second cooldown.
- Password reset token: 1 hour expiry, single-use, invalidates all existing refresh tokens on successful reset.

**Rate limits** (per IP unless noted)
- `/auth/login`: 5 requests / minute
- `/auth/register`: 3 requests / minute
- `/auth/forgot-password`: 3 requests / 15 minutes
- All other authenticated routes: 100 requests / minute per user

**Pagination, search, sort, filter (applies to every list endpoint)**
- Query params: `page` (default 1), `pageSize` (default 20, max 100), `sort` (e.g. `-createdAt`), `search`, plus resource-specific filters.
- Response includes a `meta` block: `{ page, pageSize, total, totalPages }`.

**File uploads**
- Max size: 5 MB for images, 10 MB for documents.
- Allowed types: images — `jpg/png/webp`; documents — `pdf/jpg/png`.
- Storage path convention: `uploads/{employeeId}/{category}/{uuid}-{originalFilename}`.
- Reject and return `422` on type/size violation before touching disk.

**Attendance rules**
- Working hours = checkout − checkin, computed server-side.
- Half-day threshold: configurable, default 4 hours worked.
- No check-out without a matching check-in on the same calendar day.
- No cron/scheduled auto-absent job in scope — Admin manually marks absences for employees with no check-in. Note this as a documented limitation, not a silent gap.
- All timestamps stored in UTC; display in the org's configured timezone (default `Asia/Kolkata`).

**Leave rules**
- Add a `LeaveBalance` entity (not in the original list) tracking remaining days per employee per leave type per calendar year. Initialize on employee profile creation with admin-configurable defaults (e.g. Paid: 18, Sick: 10, Unpaid: unlimited).
- Reject overlapping pending/approved leave requests for the same employee.
- Deduct balance on approval, restore on rejection or employee-cancellation of a still-pending request.
- Leave duration = inclusive calendar days between start and end date (no weekend/holiday exclusion in scope — document as a known simplification).

**Notifications — trigger list** (not specified in the original SRS; required for the module to mean anything)
- Leave submitted → notify Admin/HR
- Leave approved/rejected → notify the requesting employee
- Payroll entry generated → notify the employee
- Email verified → notify the employee
- Document uploaded/rejected → notify Admin

**Currency & locale**
- Currency: INR, stored as integer paise to avoid float rounding errors, formatted on display.

## 3. Tech Stack (pinned)

Frontend: Next.js 15 (App Router), React 19, TypeScript, Tailwind CSS, shadcn/ui, React Hook Form, TanStack Query v5, Axios, Zod.

Backend: Node.js 20 LTS, Express.js, TypeScript, ts-node-dev for local dev.

Database: PostgreSQL 16.

ORM: Prisma (local Postgres, not Prisma Accelerate/Data Platform).

Auth: JWT (`jsonwebtoken`), bcrypt (cost factor 12), `express-rate-limit`, `helmet`, `cors`.

Email: Nodemailer against a local SMTP (Mailtrap or `maildev` in Docker Compose for dev) — do not hard-depend on a paid transactional email provider.

Testing: Jest + Supertest (backend integration tests, required). React Testing Library for the auth and leave-apply flows (frontend, required); broader frontend coverage optional if time permits.

API docs: OpenAPI 3.0 via `swagger-jsdoc` + `swagger-ui-express`, served at `/api/docs`.

No Firebase, no Supabase, no BaaS, no ORM other than Prisma.

## 5. Data Model

Entities: `User`, `EmployeeProfile`, `Attendance`, `LeaveRequest`, `LeaveBalance`, `Payroll`, `SalaryStructure`, `Document`, `Notification`, `RefreshToken`.

Rules for every model:
- UUID primary keys (`@default(uuid())`)
- `createdAt`, `updatedAt` on every model
- Soft delete via `deletedAt DateTime?` on `User`, `EmployeeProfile`, `LeaveRequest`, `Document` — never hard-delete employee-linked records
- Enums for: `Role` (`EMPLOYEE`, `ADMIN`), `AttendanceStatus` (`PRESENT`, `ABSENT`, `HALF_DAY`, `LEAVE`), `LeaveType` (`PAID`, `SICK`, `UNPAID`), `LeaveStatus` (`PENDING`, `APPROVED`, `REJECTED`, `CANCELLED`), `EmploymentStatus` (`ACTIVE`, `ON_LEAVE`, `TERMINATED`)
- Index foreign keys and any column used in list-endpoint filtering (`Attendance.date`, `LeaveRequest.status`, `EmployeeProfile.department`)

Write the full `schema.prisma` before any controller code. Treat schema changes after Phase 1 as expensive — get this right early.

## 6. Auth & Security Spec

- Every protected route passes through an `authenticate` middleware (verifies access token) and, where needed, an `authorize(role)` middleware.
- Ownership checks matter, not just role checks: an Employee hitting `GET /employees/:id` or `PATCH /leave/:id` must only succeed for their own records; Admin bypasses this check.
- Input sanitization on every mutating route in addition to Zod validation (strip unexpected fields, escape for stored-XSS on free-text fields like leave `reason`/`remarks`).
- `helmet` defaults + explicit `Content-Security-Policy` for the API responses.
- CORS locked to the frontend origin from env config, not `*`.

## 7. API Contract Standard

Every response follows one envelope:

```json
// success
{ "success": true, "data": { ... }, "meta": { "page": 1, "pageSize": 20, "total": 134, "totalPages": 7 } }

// error
{ "success": false, "error": { "code": "VALIDATION_ERROR", "message": "Human-readable summary", "fields": { "email": "Invalid email format" } } }
```

Status code table:
- `200` read success, `201` create success, `204` delete success (no body)
- `400` malformed request, `401` unauthenticated, `403` authenticated but not authorized, `404` not found, `409` conflict (e.g. duplicate email, overlapping leave), `422` validation failure, `429` rate limited, `500` unhandled server error

Centralized error handler is the *only* place that formats error responses — controllers throw typed errors, they never format the response body directly.

## 8. Module Business Rules

Carried over from the original SRS with the Section 2 defaults applied:

- **Auth**: register → verify email → login (blocked until verified) → access/refresh cycle → logout (revokes current refresh token) → forgot/reset password.
- **Employee Profile**: Employee edits `phone`, `address`, `profilePicture` only; every other field is Admin-only, enforced server-side (not just hidden in the UI).
- **Attendance**: check-in/out, daily/weekly/monthly views, Admin filters by department/date/employee/status.
- **Leave**: apply/view/cancel (own), Admin approve/reject with comment, balance tracking per Section 2.
- **Payroll**: Employee read-only view; Admin edits salary structure and generates payroll entries. Generating a payroll entry is a distinct, explicit action (`POST` via `PUT /payroll/:id` semantics as an upsert, or add `POST /payroll/generate` — pick one and document it in `CONTEXT.md`).
- **Notifications**: triggers per Section 2, mark-as-read endpoint.
- **File Upload**: multer-backed, constraints per Section 2, files linked to `Document` records rather than referenced by bare path in other tables.

## 9. Frontend Requirements

- Desktop-first, responsive down to tablet width at minimum.
- Every list view: loading skeleton, empty state, pagination, search, sort, filter — wired to real query params, not client-side-only filtering once data volume matters.
- Every mutating action with irreversible consequence (delete, reject leave, terminate employee) goes through a confirmation dialog.
- Toast notifications for every mutation's success/failure.
- Global Axios instance with an interceptor that handles silent access-token refresh on 401 before failing the original request.
- TanStack Query for all server state; no server data duplicated into local component state beyond form drafts.
- Zod schemas driving React Hook Form validation, mirroring backend validators field-for-field.

## 10. Non-Functional Requirements

- **Testing**: Supertest integration test per endpoint covering happy path + one auth-failure case + one validation-failure case, minimum. RTL tests for login and leave-apply flows.
- **Logging**: structured request logging (`morgan` or `pino`) in the backend; no `console.log` left in committed code.
- **Docker**: `docker-compose.yml` with services for `postgres`, `backend`, `frontend`, and a dev SMTP catcher (`maildev`), each with healthchecks; backend waits on Postgres healthcheck before running migrations.
- **Env templates**: `.env.example` for both frontend and backend, every variable documented with a one-line comment.
- **README**: setup from clone to running app in under 10 commands, including seed data credentials for a demo Admin and demo Employee account.

## 12. Context Management Protocol

Because this build spans many phases and likely many separate sessions, context loss is the primary risk to output quality — not lack of capability.

- Before Phase 0: write `AGENTS.md` (this brief's operating rules), `CONTEXT.md` (empty, ready for decisions), `TASK.md` (Phase 0's checklist), `PROGRESS.md` (empty).
- After every phase: append decisions to `CONTEXT.md`, rewrite `TASK.md` for the next phase, update `PROGRESS.md`.
- If resuming in a new session: read all four files before writing any code.
